# 学术论文 AIGC 降重工作流

基于本科毕业论文实战验证。将知网 AIGC 检测率从 48.9% 降至 11.3%。

核心发现：**完全段落重写有效，自动化工具替换无效**。**预防性写作比事后修复高效得多**。

## 两种使用场景

1. **写新内容**（预防模式） → 直接参考"预防性写作规则"
2. **改已有内容**（修复模式） → 按"工作流"五步执行

## 预防性写作规则

写新文本时遵循以下 8 条规则，从源头避免 AI 特征。比事后重写高效 5-10 倍。

### 规则1：用具体动词，不用抽象动词

| 避免 | 用这些替代 |
|------|-----------|
| 进行了…分析 | 分析了 / 做了…分析 |
| 实现了…功能 | 做了 / 完成了 |
| 采用…方法 | 用了 / 选了 |
| 开展了…研究 | 做了…研究 |
| 存在…问题 | 有…问题 |

### 规则2：长短句交替，打破均匀节奏

AI 写作的典型特征是每句长度差不多（20-30字）。人类写东西是长短不均的。

```
差：系统采用三层架构设计。前端使用Vue 3框架开发。后端使用FastAPI框架实现。
    每句都是12字，节奏机械。

好：系统用三层架构。前端Vue 3，后端FastAPI，中间Tauri桌面壳负责窗口管理和进程调度。
    第二句明显更长，节奏自然。
```

### 规则3：不编号，自然过渡

```
差：第一类走统计路线…第二类走网络分析路线…第三类是深度学习方法…
好：国内外研究者从不同角度切入，思路大致有三类。一类走统计路线…另一类做网络分析…也有用深度学习的…
```

### 规则4：直接说事，不做铺垫

```
差：在中药现代化研究的背景下，随着高通量筛选技术的发展，越来越多的研究者开始关注…
好：中药成分关系复杂，传统统计方法搞不定，研究者开始尝试新的技术路线。
```

### 规则5：去连接词，用标点代替

```
差：此外，系统还支持多种预设配置。与此同时，用户可以自定义参数。
好：系统支持多种预设配置，也允许用户自定义参数。
```

### 规则6：段落长度不均匀

不要每段都 4-5 句。有的段落 2 句话（快速带过），有的 5-6 句（展开讨论）。像人写东西时自然的长短变化。

### 规则7：用"而""则"做对比，不用"不仅…而且…"

```
差：该系统不仅能够发现成分间的关联关系，而且能够自动推导出数学表达式。
好：系统能发现成分间的关联关系，而符号回归引擎把这些关系写成显式公式。
```

### 规则8：学术文本中偶尔用短句穿插

在严谨的学术段落中穿插 10 字以内的短句，打破节奏但不失严肃性。

```
好：技术可行性分析表明，PyTorch和gplearn能够支撑模型训练与符号回归环节，Tauri、
    Vue 3和FastAPI的组合满足跨平台桌面应用的开发需要，且均为开源工具，经济成本可忽略。
    （最后9字的短句"经济成本可忽略"打破节奏）
```

## 核心原则

1. **完全重写，不要逐词替换** — 摘要从 98.3% 降到 0% 靠的是整段重写
2. **只改 w:t 文本节点，不动格式** — 用 lxml + zipfile 直接操作 OOXML
3. **按报告定位，按优先级改** — 先改字符数最多的"显著"片段
4. **跨报告交叉比对** — 多次检测报告中重复出现的段落优先处理

## 工作流

### 第一步：解析检测报告

从 CNKI AIGC 全文报告中提取信息：

1. 提取各章节的 AI 特征值和字符数
2. 标记所有"AI特征显著"（计入 AI 率）的片段
3. 按 AI 字符数排序确定改写优先级

报告结构：每个片段有序号、字符数、AI特征等级（显著/疑似）。**只有"显著"计入 AIGC 率**。

### 第二步：定位目标段落

对每个"显著"片段：
1. 从报告中复制片段原文
2. 在 docx 中用唯一子串（search_key）定位对应段落
3. 确认 old_text 完全匹配（注意前次改写可能已改变文本）

### 第三步：重写段落

**有效的重写策略**（按重要性排序）：

| 策略 | 说明 | 示例 |
|------|------|------|
| 打破三段式 | "首先…其次…最后" 改为自然过渡 | 不编号，用逻辑连接 |
| 去模板句式 | "在…方面" 每段出现 3+ 次 | 直接说事，不做话题铺垫 |
| 缩短长句 | 40+ 字的句子拆开 | 在转折词处断开 |
| 注入具体性 | 抽象概括换为具体数据 | "效果很好" → "R²从0.72升到0.81" |
| 变化节奏 | 长短句交替 | 不要每句都 20-30 字 |
| 去连接词 | "此外""同时""因此" 删或换 | 用句号代替连接词 |

**必须避免的 AI 高频词**：此外、与此同时、综上所述、值得注意的是、具有显著优势、具有重要意义、不仅…而且…、其一…其二…其三…

**重写风格**：
- 保持学术严谨性，不口语化
- 用短句、直接陈述
- 保留所有技术术语和引用标记
- 保留数字、参数、公式不变

### 第四步：应用替换

使用 [scripts/docx_rewrite.py](scripts/docx_rewrite.py) 操作 docx：

```python
replacements = [
    (search_key, old_text, new_text),
    ...
]
```

关键约束：
- 先读所有文件到内存 dict，修改后写临时文件再 os.replace
- replace_in_paragraph 处理跨多个 w:r 元素的文本
- 每个批次修改后验证文件完整性

### 第五步：验证

1. 检查文件大小合理（不应剧变）
2. 检查总字符数合理（重写后通常略减少）
3. 检查无文本重复（如 "ML后端ML后端"）
4. 重新提交检测验证实际降幅

## 跨报告交叉比对策略

当有多份检测报告时：
1. 对比两份报告中都标记为"显著"的片段 → 优先改写
2. 只在第二份报告中新增的"显著"片段 → 次优先
3. 在第一份中标记但第二份中消失的 → 已有效，跳过

## 批次执行策略

按 AI 字符数从大到小分批处理：

| 优先级 | 章节 | 原因 |
|--------|------|------|
| P0 | 摘要 | 占比大，完全重写效果最好 |
| P1 | AI率 >50% 的章节 | 投入产出比最高 |
| P2 | AI率 30-50% 的章节 | 中等优先 |
| P3 | AI率 <30% 的章节 | 低优先，性价比低 |

每个批次 10-20 个替换，执行后立即验证。避免单次替换超过 25 个（语法错误风险）。

## 不工作的方法

| 方法 | 为什么不行 |
|------|-----------|
| humanize-chinese 自动改写 | 插入古文、错误替换、产生新的 AI 特征 |
| 逐词替换 AI 高频词 | 只改词汇不改结构，检测器仍标记 |
| python-docx 库操作 | 会重写整个 XML，丢失复杂格式 |
| 只改"疑似"片段 | "疑似"不计入 AI 率，改了也不降分 |
